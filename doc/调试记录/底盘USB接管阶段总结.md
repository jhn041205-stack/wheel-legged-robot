# 底盘 USB 接管阶段总结

## 1. 本阶段目标

本阶段的核心目标是把底盘控制链路从“仅遥控器控制”扩展为“遥控器授权 + USB 接管控制”的模式，并保证：

- 前期可以继续使用实体遥控器完成移动与部署。
- 到达目标位置后，可以通过实体遥控器的指定开关授权 USB 接管。
- USB 接管后，底盘最终只接收 USB 生成的控制量，不与遥控器摇杆量混合。
- 代码层面形成清晰、可维护、可继续调试的控制闭环。

## 2. 当前已经达成的阶段成果

### 2.1 控制源已经完成拆分

目前控制链路中已经形成三个层次：

- `raw_rc_ctrl`
  - 保存实体遥控器接收到的原始控制量。
  - 仅用于遥控器本体控制以及 USB 授权判断。

- `VIRTUAL_RC_CTRL`
  - 保存 USB 侧生成的虚拟遥控器控制量。
  - 当前由 `0x01` 机器人控制包转换得到。

- `rc_ctrl`
  - 这是底盘真正读取的活动控制量。
  - 每次只会从 `raw_rc_ctrl` 和 `VIRTUAL_RC_CTRL` 里二选一复制。

### 2.2 USB 授权逻辑已经明确

当前 USB 接管条件为：

- 实体遥控器 `CHASSIS_MODE_CHANNEL == DOWN`
- USB 在线

只有同时满足这两个条件时，活动控制量 `rc_ctrl` 才切换为 `VIRTUAL_RC_CTRL`。

否则：

- 活动控制量 `rc_ctrl` 始终保持为实体遥控器的 `raw_rc_ctrl`。

因此当前设计不是“USB 和遥控器并行混控”，而是“遥控器授权后，控制权切换到 USB”。

### 2.3 USB 的 0x01 数据包已经接入到底盘 RC 控制语义

当前 `usb_task.c` 已完成如下处理：

- 接收上位机 `0x01` 控制包
- 解析为 `ROBOT_CMD_DATA`
- 将其中的速度量转换为虚拟遥控器通道量 `VIRTUAL_RC_CTRL`
- 再通过活动控制仲裁进入底盘

当前转换语义为：

- `s[CHASSIS_MODE_CHANNEL] = RC_SW_MID`
  - 让底盘处于正常平衡移动模式

- `s[CHASSIS_FUNCTION] = RC_SW_DOWN`
  - 与当前底盘控制框架的常规移动模式保持一致

- `ch[CHASSIS_X_CHANNEL] <- vx`
  - 前后速度

- `ch[CHASSIS_WZ_CHANNEL] <- -wz`
  - 旋转角速度

- `ch[CHASSIS_TAIL_POS_CHANNEL] = 0`
  - 避免 0x01 数据包缺少尾部/附加控制量时引入非预期尾部动作

### 2.4 USB 发状态链路保持独立

本阶段没有破坏已有的 USB 状态上报链路。

也就是说：

- 下位机通过 USB 向上位机发送状态，仍可继续工作
- 上位机通过 USB 向下位机发送 `0x01` 控制包，已经有明确的接入路径

### 2.5 代码结构已经做过一次清理

为了让后续控制调试更可靠，本阶段还整理了以下代码问题：

- 清理了被注释吞掉的宏定义和重复定义
- 修复了 `usb_task.c` 中被污染的接收缓存处理代码
- 修复了 USB 初始化处重复残留的 `sizeof(...)`
- 清理了 `remote_control.h`
- 整理了 `robot_param.h` 的入口宏定义

这一步的意义不是功能新增，而是把“逻辑大体正确但代码形态脏乱”的状态，整理为“可继续稳定调试”的状态。

## 3. 当前控制链路的实际工作方式

### 3.1 实体遥控器工作阶段

当实体遥控器未拨到 USB 授权位，或 USB 离线时：

- 串口接收实体遥控器数据
- 更新 `raw_rc_ctrl`
- 活动控制量 `rc_ctrl = raw_rc_ctrl`
- 底盘只受实体遥控器控制

### 3.2 USB 接管阶段

当实体遥控器拨到授权位，且 USB 在线时：

- USB 接收 `0x01` 控制包
- 转换为 `VIRTUAL_RC_CTRL`
- 活动控制量 `rc_ctrl = VIRTUAL_RC_CTRL`
- 底盘只读取 USB 虚拟遥控器控制量

此时实体遥控器仍然有一个作用：

- 决定是否继续授权 USB 接管

但实体遥控器的摇杆量本身不再参与底盘运动控制。

## 4. 本阶段确认过的重要结论

### 4.1 当前不存在“USB 与遥控器摇杆量同时叠加控制底盘”的逻辑

原因是底盘最终只读取一个活动结构 `rc_ctrl`，而这个结构每次仲裁只会选择一个来源：

- 选实体遥控器
- 或选 USB 虚拟遥控器

所以这是“切换控制源”，不是“多源混合控制”。

### 4.2 USB 虚拟遥控器中的 `s[1] = MID` 不适合作为 USB 授权依据

这一点在方案讨论中已经明确：

- USB 虚拟遥控器本身是为了让底盘进入正常平衡移动模式
- 它的 `s[1]` 只是虚拟输入内容的一部分
- 如果直接依据活动控制量中的 `s[1]` 再做授权判断，会产生语义混乱

因此当前采用的是更合理的方式：

- 只用实体遥控器原始输入 `raw_rc_ctrl.rc.s[CHASSIS_MODE_CHANNEL]`
- 作为 USB 是否接管的授权开关

### 4.3 0x01 数据包目前只覆盖了“平衡移动”最基本的控制量

目前 0x01 转换后的虚拟 RC 主要覆盖：

- 前后移动
- 原地转向
- 正常平衡移动模式选择

尚未覆盖更多扩展控制量，例如：

- 侧向或更复杂姿态控制
- 夹爪/机械臂类附加功能
- 更复杂的开关态行为

所以当前版本适合作为“USB 接入和基本移动验证”的稳定基础版。

## 5. 当前仍需关注的问题

### 5.1 当前更适合做“控制调试”，还不适合直接视为最终版本

虽然链路已经闭环，但下一阶段仍需要重点验证：

- `vx` 映射后前进方向是否与实车一致
- `wz` 的正负方向是否与期望一致
- USB 接管切换瞬间是否平顺
- USB 掉线后是否能可靠退回遥控器控制
- 在授权位保持不动时，底盘是否会受到旧 USB 数据残留影响

### 5.2 存在一个并发层面的实现风险

当前 `rc_ctrl` 会被不同执行上下文刷新：

- 遥控器串口中断
- USB 任务

而底盘任务会读取它。

从控制语义看这没有问题，但从实现细节看，理论上仍存在非常小的并发一致性风险。这个问题更偏“代码鲁棒性优化”，不是当前控制逻辑的主矛盾。

## 6. 建议的下一阶段调试重点

下一阶段建议集中处理“控制调试问题”，按以下顺序推进：

1. 先验证 USB 授权切换是否稳定
   - 遥控器正常控制
   - 拨到授权位后切到 USB
   - USB 掉线后退回遥控器

2. 再验证 0x01 到底盘运动的方向与比例
   - `vx` 正负
   - `wz` 正负
   - 通道比例与速度手感

3. 最后再做控制体验优化
   - 切换时防突变
   - 限幅和缓启动
   - USB 无新包时的降级策略

## 7. 当前阶段结论

本阶段已经完成的不是“最终控制优化”，而是更重要的基础工作：

- 把 USB 控制正式接入到底盘控制链路
- 把 USB 授权逻辑放在实体遥控器原始输入上
- 避免 USB 与实体遥控器同时混控
- 形成可以继续调试的干净代码基础

因此，下一阶段的重点已经可以从“链路是否能通”转为“控制是否顺手、是否稳定、是否安全”。
